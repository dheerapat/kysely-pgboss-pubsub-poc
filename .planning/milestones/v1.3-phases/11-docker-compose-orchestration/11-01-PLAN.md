Create `docker-compose.yml` at the project root with the postgres service. Use the exact structure below:

```yaml
services:
  postgres:
    image: postgres:17
    environment:
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: postgres
    volumes:
      - postgres-data:/var/lib/postgresql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U admin -d postgres"]
      interval: 5s
      timeout: 5s
      retries: 5
      start_period: 10s

volumes:
  postgres-data:
    driver: local
```

Key decisions:
- Service name is `postgres` (not `postgres-db`) — this is the DNS name used by app replicas in `DATABASE_URL`
- `postgres:17` pinned per STATE.md (not `latest` — pg-boss 12.5.4 compatibility with pg18 untested)
- Credentials match `docker-compose.postgres.yaml` (admin/pass/postgres) — same data layer, separate orchestration
- Volume path `/var/lib/postgresql` matches existing file (not `/var/lib/postgresql/data` — intentional)
- No `ports:` on postgres — only app needs to reach it, via internal Compose network
- `start_period: 10s` — gives postgres time for initial cluster setup before healthcheck retries count
- `retries: 5` with `interval: 5s` — app replicas wait up to ~35s for postgres (start_period + 5×5s) before failing

  <action>
Add the `app` service to `docker-compose.yml`. The final file must contain BOTH services. Add the `app` service block under `services:`:

```yaml
  app:
    build: .
    environment:
      DATABASE_URL: postgres://admin:pass@postgres:5432/postgres
      PGBOSS_MAX_CONNECTIONS: "5"
    depends_on:
      postgres:
        condition: service_healthy
    deploy:
      replicas: 6
```

Key decisions:
- `build: .` uses the Dockerfile at project root (Phase 10 output)
- `DATABASE_URL` points to `postgres` service name (not `localhost`) — this is the critical wiring; `localhost` inside Docker resolves to the container loopback, causing ECONNREFUSED
- `PGBOSS_MAX_CONNECTIONS: "5"` — env var for pg pool max; app code must read this (see acceptance criteria note below)
- `depends_on: condition: service_healthy` — replicas only start after postgres passes its `pg_isready` healthcheck; prevents crash-loop / connection-refused race condition on startup
- `deploy.replicas: 6` — starts 6 identical app instances; pg-boss multi-master is safe (uses `pg_advisory_xact_lock()`)
- NO `ports:` block on app service — replicas are accessible only from within the Compose network; Caddy (Phase 12) will be the sole host-facing port

**Important — pg pool max env var:** The app currently hardcodes `new Pool({...})` without a `max` option. After writing the Compose file, also update `src/infrastructure/db/pool.ts` to read `PGBOSS_MAX_CONNECTIONS`:

```typescript
// src/infrastructure/db/pool.ts
import { Pool } from "pg";

export const pool = new Pool({
  connectionString: process.env["DATABASE_URL"] ??
    "postgres://admin:pass@localhost:15432/postgres",
  max: parseInt(process.env["PGBOSS_MAX_CONNECTIONS"] ?? "10", 10),
});
```

This satisfies COMP-03: pg pool `max: 5` via env var (6 × 5 = 30 connections, under Postgres default 100).
