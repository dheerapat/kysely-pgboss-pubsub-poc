Update README.md to document the v1.1 fan-out pub/sub model. Add a new section titled "## Pub/Sub Fan-Out (v1.1)" immediately after the existing "## The Core Thesis" section. The new section must contain all four of these explanations:

**1. Pub/Sub vs Queue-Based Approach**

Explain that the v1.0 model used `boss.send(queueName)` — a direct point-to-point queue send requiring the publisher to know every consumer queue. The v1.1 model uses `boss.publish(eventName)` — a pub/sub channel broadcast where the publisher only names the event; pg-boss resolves fan-out targets from the `pgboss.subscription` table. This decouples publishers from consumers: adding a new subscriber requires no publisher code changes.

**2. pgboss.subscription Table Role**

Explain that `pgboss.subscription` is pg-boss's internal routing registry. Each row maps an event channel name (e.g. `"user.registered"`) to a subscriber queue name (e.g. `"notification.user.registered"`). When `boss.publish("user.registered", payload)` is called, pg-boss queries this table to discover all target queues and INSERTs one job per queue. The table is populated by calling `boss.subscribe(eventName, queueName)` during boot.

**3. Fan-Out Mechanism**

Explain the fan-out sequence: (1) `boss.publish("user.registered", payload, { db: tx })` is called with the active Kysely transaction; (2) pg-boss SELECT-queries `pgboss.subscription` to find all registered queues for `"user.registered"` — currently `notification.user.registered` and `audit.user.registered`; (3) pg-boss INSERTs one job into each queue using the provided `db` transaction adapter; (4) both INSERTs commit or roll back atomically with the domain INSERT.

Include this code illustration:
```
boss.publish("user.registered", payload, { db: tx })
  └─ pgboss.subscription lookup
       ├─ → INSERT job into notification.user.registered (via tx)
       └─ → INSERT job into audit.user.registered (via tx)
  // If tx rolls back → BOTH job INSERTs roll back → zero orphaned jobs
```

**4. Boot Sequence Ordering Rationale**

Explain that the boot order `createQueue → boss.subscribe → boss.work → app.listen()` is enforced because: `pgboss.subscription` has a foreign key on queue names, so the queue must exist before `boss.subscribe()` can register it; `boss.publish()` silently produces zero jobs if no subscriptions are registered at publish time; therefore ALL subscriptions must complete before `app.listen()` to guarantee that the first HTTP request does not race against subscription setup.
